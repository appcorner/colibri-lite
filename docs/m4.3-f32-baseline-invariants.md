# M4.3 Frozen F32 Baseline Invariants

## Purpose

This checklist gates every implementation compared with the M4.3-01 baseline.
It does not authorize optimization. The active task must separately permit the
proposed dtype, kernel, cache, I/O, or accelerator change.

## Mandatory Identity and Semantic Invariants

- [ ] Model ID, revision, architecture, canonical manifest hash, tensor names,
  shapes, offsets, orientations, and packed expert order retain their frozen
  identity unless a separately versioned artifact contract is reviewed.
- [ ] Router ordering remains higher score first and lower expert ID first for
  exact ties.
- [ ] Exact structural evidence preserves token, layer, expert, projection,
  routing-weight, and aggregation membership.
- [ ] Tier A input `[9707, 11, 1879, 0]` generates `[1096, 374]`.
- [ ] Required F32 router and token decisions remain safely separated under
  the scoped margin rules.
- [ ] Tier B guard router IDs, top-20 ranks, and argmax IDs retain their frozen
  evidence or are explicitly classified by a new reviewed contract.
- [ ] Guarded outputs contain no NaN or infinity.
- [ ] KV cache positions advance once, prior positions are not overwritten,
  primary decode remains cached, and allocation stays bounded.
- [ ] Expert-cache residency stays within the declared byte budget and cache
  counters reconcile.
- [ ] Evidence remains readable under the frozen schema or uses an explicit
  new schema version.

## Numerical Invariants

- [ ] The ordered scalar Rust F32 baseline remains unchanged and reproducible.
- [ ] Exact internal contracts and the provisional PyTorch RMSNorm guard retain
  their M4.2 scopes.
- [ ] Tier B final-norm and compact-logit budgets remain fixture-specific; they
  are not promoted to universal tolerances.
- [ ] F64 evidence remains diagnostic-only.
- [ ] A lower-precision or different-accumulation implementation declares its
  own comparison contract without weakening the F32 baseline.
- [ ] The first failing checkpoint is diagnosed before later output is
  accepted.

## Required Variant Deltas

Every proposed variant reports:

- [ ] correctness delta;
- [ ] numerical-contract delta;
- [ ] memory and working-set delta;
- [ ] logical and, when valid, physical I/O delta;
- [ ] startup, prefill, and decode latency/throughput delta;
- [ ] artifact-size delta;
- [ ] platform, dependency, unsafe, and toolchain requirements.

## Tradeable Metrics

Working set, artifact size, logical bytes read, startup time, throughput, and
cache hit rate are tradeable only after mandatory correctness invariants pass.
No improvement in a tradeable metric compensates for a correctness failure.

## Regression Selection

- [ ] Run standard workspace, warning-free Clippy, Python compilation, bundle
  reproducibility, evidence-hash, and schema tests for every change.
- [ ] Run relevant Tier C operation guards during focused development.
- [ ] Run Tier B for changes that affect full-layer arithmetic, routing,
  experts, final norm, LM head, token handling, or KV updates.
- [ ] Run Tier A before accepting quantization, cache/I/O redesign, SIMD,
  alternative accumulation, parallel experts, or accelerator work.
- [ ] Re-measure resource metrics using the same fixture and clearly state
  filesystem-cache and logical-versus-physical-I/O assumptions.

## Stop Conditions

Stop for review when an exact invariant changes, a budget must be widened, a
safe selection becomes ambiguous or mismatched, evidence becomes incompatible,
or a new unsafe/dependency/artifact/public contract is required.
