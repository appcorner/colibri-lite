# ADR 0040: Representative Full-Runtime 8/16 GiB Cache Validation

## Status

Accepted with workload limitations.

## Context

M5.2-02 simulated strict global LRU over eight representative ordered expert
traces and selected exact 8 GiB versus 16 GiB as the minimal full-runtime
matrix. The simulation classified 8 GiB as useful for selected workloads, but
the short Tier-A runtime fixture was insufficient to establish workload
generality. This task must validate cache counters, correctness, bounded
residency, logical-read reduction, process memory, and timing without changing
the runtime cache policy or numerical path.

## Decision

Retain strict global LRU and expose only explicit payload-byte configuration.
Treat 8 GiB as a workload-selective mode and 16 GiB as a performance-oriented
mode for cacheable workloads; neither is a universal production preset. Do not
implement auto-sizing or another cache policy in this task.

The cache primitive remains `accepted_with_workload_limitations`. A future
workload-aware budget design may distinguish no-reuse short sessions from
repeated long-context/control sessions. The next optimization study is
mmap/coalesced expert access, not a cache-policy change.

## Evidence

- Six selected fixtures were run at both exact budgets, 18 runs total.
- Tier-A, long-context, and long-decode were repeated twice per budget.
- All runtime counters matched the M5.2-02 exact-budget simulation fields.
- All generated IDs, request traces, finite outputs, KV invariants, and bounded
  payload checks passed; oversized and blocked-eviction events were zero.
- Selected-subset macro/micro byte hit rates were 19.1488%/27.3838% at 8 GiB
  and 24.8884%/36.1178% at 16 GiB.
- Thai and special-token workloads had zero hits at both budgets; long context
  was the best case at 31.9155% and 45.0810% byte hit rate.
- Peak expert payload was 8,587,837,440 bytes at 8 GiB and 17,175,674,880
  bytes at 16 GiB. Peak process working set was measured separately.

## M5.1-03 discrepancy resolution

The earlier M5.1-02 replay charged modeled metadata/alignment overhead before
assigning payload capacity. Its usable payload budgets were 8,212,550,504 and
16,802,485,096 bytes, while the runtime and M5.2-02 use the exact nominal
payload budgets. That explains the 8 GiB eviction difference and the 16 GiB
hit/load/eviction difference. The request boundary and lease lifetime were not
the cause, and no runtime semantic defect was found.

## Measurement contract

The runtime-validation instrumentation adds timing, phase, process-memory,
and machine-readable cache/I/O metrics around the existing execution. It does
not alter cache insertion, promotion, eviction, budget accounting, router
execution, or arithmetic. The additive result field
`simulation_comparison` records exact agreement for the simulation fields.
Canonical ordered traces remain the identity for request-order comparison.

## Limitations

Filesystem cache state was uncontrolled; timing is directional and no physical
I/O or token/s claim is made. Persistent cross-session caching was not run in
the full runtime. Long-context and long-decode fixtures are deterministic Rust
regression/trace fixtures rather than new Transformers-equivalence claims.
Cache metadata is not separately instrumented; only a structural lower-bound
estimate from resident entry count is reported.
