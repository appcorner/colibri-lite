# ADR 0020: M4.2 Numerical Validation Budgets

- Status: Accepted provisionally for M4.2 layer 0
- Date: 2026-07-15
- Milestone: M4.2
- Task: M4.2-02
- Source revision: `ad44e777bcd18fa416d9da3bd8f70d33ebb85d39`

## Context

The Rust scalar RMSNorm path and an ordered Python F32 implementation match
bit-for-bit for every scalar intermediate and all 8,192 layer-0 outputs. A
one-thread deterministic PyTorch F32 control differs because its reduction tree
is an internal implementation detail and not a stable public contract. The
isolated same-input comparison observed at most 5 ULP and
`3.5762786865234375e-7` absolute output error.

The earlier end-to-end F32 checkpoint guard mixed two different validation
questions: whether Rust is internally deterministic and whether independent
runtimes using different reduction orders remain numerically equivalent. A
single absolute/relative tolerance cannot express both questions faithfully.

## Decision

M4.2 uses two validation modes.

### Internal deterministic validation

When the reference executes the documented Rust scalar operation order with
F32 storage after each operation, scalar intermediates and outputs must match
Rust exactly. Integer IDs, shapes, indices, and tensor names also remain exact.
No tolerance is permitted for this mode.

### Cross-runtime validation

Independent runtimes may use different reduction trees. Reduction checkpoints
therefore use both an ULP guard and an absolute numerical-error guard. This does
not authorize changes to Rust arithmetic or imitation of PyTorch internals.

For isolated RMSNorm with identical F32 inputs and weights:

```text
observed maximum ULP error      = 5
provisional maximum ULP guard   = 8
observed maximum absolute error = 3.5762786865234375e-7
provisional absolute guard      = 5.0e-7
```

The extra 3 ULP and `1.4237213134765625e-7` are a small provisional allowance
for platform/compiler variation while remaining close to the measured layer-0
result. Both guards must pass.

End-to-end reduction checkpoints use a propagated budget rather than raising a
legacy tolerance. For layer-0 post-attention RMSNorm:

```text
incoming residual maximum error       = 1.2516975402832031e-6
incoming-error safety factor           = 3.0
isolated RMSNorm absolute guard         = 5.0e-7
propagated maximum absolute budget      =
    3.0 * 1.2516975402832031e-6 + 5.0e-7
  = 4.255092620849609e-6
observed end-to-end maximum error       = 3.993511199951172e-6
```

The observed effective amplification after subtracting isolated reduction
variance is approximately `2.9047619`; factor 3 adds narrow headroom without
claiming a general RMSNorm Lipschitz bound. This budget is checkpoint-specific
and cannot be copied to another layer or operation without new measurements.

Router expert IDs follow ADR 0019. Exact cross-runtime IDs are assertable only
when the top-k boundary margin is safely positive and exceeds at least twice
the measured router-logit error. Ambiguous boundaries are reported, not failed.

## Provisional status

These guards are provisional for M4.2. Layers 1, 24, and 47 must independently
record incoming error, isolated reduction variance where applicable, propagated
checkpoint error, and top-k margins. The guards and safety factor must then be
re-evaluated; they do not become a general runtime promise by surviving layer 0.

## Consequences

- Rust RMSNorm, router math, runtime F32 dtype, and deterministic sorting remain
  unchanged.
- Internal scalar regressions remain exact and cannot be hidden by a
  cross-runtime budget.
- Cross-runtime evidence records the reason and components of each propagated
  budget.
- No public model, artifact, or execution API changes.
