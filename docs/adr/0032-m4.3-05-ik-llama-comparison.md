# ADR 0032: M4.3-05 External ik_llama Comparison

## Status

Accepted as a measurement record; no runtime change authorized.

## Decision

Pin `ik_llama.cpp` commit `1fddd12ba861c4815a8633f14d9c5670692099cc` and use
Qwen3-30B-A3B `Q4_K_M` as the practical CPU reference. The comparison is
directional because the model format is quantized and the colibri reference is
the canonical F32 artifact. No ik_llama source is vendored and no colibri
runtime behavior is changed.

The first colibri optimization experiment should be a separately measured
resident-weight or buffered-I/O design on the F32 path. It must preserve the
M4.2/M4.3 correctness invariants before any quantized runtime is reconsidered.

## Evidence

The exact token fixture is `Hello, world!` -> `[9707, 11, 1879, 0]`. Four fresh
processes and one 32-token decode completed with finite output and exit status
zero. `llama-bench` reported 3.968 tok/s prompt and 2.819 tok/s generation for
three repetitions. Peak sampled working set was 4.36-4.41 GB for short runs and
9.03 GB for the long decode. The colibri M4.2-05 reference remains 0.0147
prompt/decode tok/s with 145-148 MB peak working set and 127,823,000 modeled
explicit bytes.

These values are not a controlled quality comparison: the Q4 model is not
numerically equivalent to colibri F32, and physical disk reads were not
measured. They establish an external architecture reference for quantized
weights, mmap/page-cache residency, fused MoE, SIMD, and threaded kernels.

## Consequences

The hardware can run useful Qwen3-30B-A3B decode speed with an optimized CPU
runtime, but colibri's scalar disk-streaming path leaves a large directional
speed gap. The low-memory distinction remains real: colibri's measured process
working set is much smaller, while ik_llama retains a large mapped model working
set. Future reports must separate controlled same-hardware findings from format
and implementation inferences.
