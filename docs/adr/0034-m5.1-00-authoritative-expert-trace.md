# ADR 0034: Capture the Authoritative Ordered Expert Trace

## Status

Accepted for M5 measurement; does not modify the M4 release.

## Context

The M4 resource baseline committed aggregate expert counters and reuse
statistics but not the ordered request sequence needed for cache simulation.
Reconstructing order from aggregate router summaries would be non-authoritative.

## Decision

Capture requests at the `ExpertStore` load boundary during an exact replay of
the frozen M4 Tier-A fixture. `load_with_observer` delegates to the existing
cache/load implementation and reports metric deltas after acquisition. The
streaming diagnostic path records each actual occurrence in runtime order.
Records are accumulated synchronously in memory and written after execution
using a fixed, compact JSON serialization.

The canonical artifact is
`models/qwen3-30b-a3b/m5.1-00-ordered-expert-trace-v1.json`, with its
manifest and validator stored beside the M4 evidence. The trace is classified
as an M5 measurement supplement replaying the frozen M4 configuration.

## Evidence

Two identical replays produced SHA-256
`f3f87f05d15424030c9261cdf3e93bd72e9c006a55303bc0c28a92a4fb3ff2d0` and
2,304 records. Validation reproduces 1,332 unique keys, zero hits, 2,304
loads, 2,303 evictions, 43,486,543,872 expert bytes, and the M4 reuse-distance
distribution. Generated IDs remain `[1096, 374]` and KV-cache contracts pass.

## Consequences

M5.1-01 can simulate deterministic cache policies from actual request order.
The trace does not claim physical device I/O, throughput improvement, or a
production cache policy. Full-vocabulary temporary logits are not retained;
the replay explicitly records that limitation while preserving committed
fixed/top-20, intermediate, guard-router, finite-output, and generated-ID
checks.
