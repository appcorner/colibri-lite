# ADR 0043: M5.3 compute-profile prototype selection

- Status: accepted for next-task review; prototype not implemented
- Date: 2026-07-18
- Scope: M5.3-03 full-model compute profiling

## Context

M5.3-02 established that the reusable staging-buffer reader is correct and
diagnostically useful but must remain feature-gated: it was slower in 9 of 12
paired full-runtime comparisons. M5.3-03 added a safe, feature-gated,
hierarchical profiler around the existing reference reader and strict global
LRU path.

The selected matrix was Tier-A control, code, long context, and long decode at
exact 8 GiB and 16 GiB budgets, plus disabled/coarse/detailed Tier-A 8 GiB
mode runs. All 10 rows preserved generated IDs, request traces, cache
counters, KV invariants, and bounded residency; simulation comparisons were
exact.

## Evidence

The detailed profiles measured `cache.lookup_and_expert_load` at 71.6–76.4%
of model-profile time. Expert MLP was 4.1–5.5%, LM head 2.8–3.8%, and
attention approximately 2.0–2.8%. The profile model-total/runtime-total ratio
was 0.99988–1.00043. The reference reader had one open, seek, read, and
allocation per expert miss; payload SHA-256 verification was a large part of
the measured logical reader path. Filesystem cache state was uncontrolled, so
these are directional runtime measurements, not physical-I/O or throughput
claims.

## Decision

Select exactly one next prototype: **isolated read-only mmap expert access**.

The prototype must use the same exact payload ranges as the reference reader,
remain opt-in and outside the production ExpertStore path, and report mapping
initialization, access, verification, copy, unmapping, virtual mapped bytes,
resident working set, and committed/private bytes separately. It must not
change the artifact layout, strict global LRU, payload budgets, leases,
request ordering, arithmetic, or default reader.

No compute kernel optimization is selected at this point because expert MLP
and LM-head shares are materially smaller than the measured storage path.
Threading, SIMD, fusion, resident dense weights, and further buffer work remain
unimplemented alternatives.

## Review gates

The next prototype may proceed only if it demonstrates:

1. exact payload and tensor byte equivalence, deterministic output, identical
   router/expert order, cache counters, logical-read counters, KV state, and
   numerical checkpoints;
2. safe Windows file/mapping handle lifetimes, alignment, unmapping,
   replacement/deletion behavior, and bounded mapping state;
3. explicit separation of virtual address space, resident working set,
   committed/private memory, logical reads, and unknown physical I/O;
4. a material reduction in the measured reader-path cost without an
   end-to-end regression outside documented timing noise;
5. no unsafe code unless unavoidable, isolated, justified, and separately
   reviewed.

The prototype must not be called production-ready or made default as part of
M5.3-03.

## Consequences

This decision directs the next experiment at the largest measured affected
share while keeping the current correctness path unchanged. It carries
Windows page-fault, working-set, mapping-lifetime, antivirus, and storage
device risks. A negative result is useful: it would support stopping storage
path work and returning to a compute candidate with measured gates.
