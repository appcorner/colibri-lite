# ADR 0044: M5.3 isolated mmap expert access

- Status: rejected for runtime adoption; retained as isolated evidence
- Date: 2026-07-18
- Scope: M5.3-04 read-only expert-shard mmap prototype

## Context

M5.3-03 measured the existing runtime as primarily storage-bound. The
reference reader performs one open/seek/read/allocation per expert miss, while
the reusable staging-buffer prototype only had microbenchmark value. This task
tested whether read-only mapping could remove syscall overhead without changing
expert ownership, strict global LRU, payload budgets, artifact layout, or
numerical execution.

## Decision

Use a separate `clr-mmap` crate with `memmap2 = 0.9.11`. It lazily maps a
complete expert shard read-only, retains the backing file handle and mapping
together, exposes only checked shared ranges, and copies the range into owned
expert storage before cache admission. Mappings are retained for one reader's
lifetime; no production mapping eviction policy is introduced. The reference
reader remains the default and mmap is only compiled/selected explicitly.

The single unsafe mapping call is isolated in `clr-mmap`, with documented
file-length, lifetime, read-only, and aliasing invariants. No unsafe code is
added to `clr-storage` or the default runtime path.

## Evidence

Byte-equivalence, lifecycle, cleanup, deterministic trace, cache, KV, and
bounded-payload gates passed. The complete 16-run full-runtime matrix matched
simulation exactly. Mmap was slower in all eight paired comparisons, with a
median total-runtime regression of 5.92%:

* 8 GiB: +2.63% to +9.33% by workload;
* 16 GiB: +5.21% to +15.32% by workload;
* mapped virtual bytes: 108 GiB in selected full-runtime workloads;
* mmap peak working set: 29.46--39.00 GiB;
* reference peak working set: 8.72 GiB at 8 GiB cache and 17.31 GiB at
  16 GiB cache.

## Consequences

Mmap removes reference open/seek/read calls from instrumented counters, but it
does not remove SHA-256 verification or the copy into owned expert storage.
Page-fault and working-set behavior add operational risk under complete-shard
mapping. The prototype is classified `insufficient_runtime_value`; mmap is
not default, optional production, or a selected bounded-mapping follow-up.

The next task is to stop the current storage-access optimization path due
insufficient runtime value. Any later storage work requires a new measured
proposal and review; compute optimization is not started by this ADR.
