# ADR 0039: M5.2 Corpus Cache-Policy and Budget Simulation

## Status

Accepted for simulation evidence; runtime policy unchanged.

## Context

M5.2-01 expanded the authoritative ordered expert-request corpus from the
short Tier-A control to eight deterministic workloads. The corpus contains
11,520 expert occurrences, 3,148 union layer/expert keys, and both short and
long prompt/decode behavior. A policy and budget decision based only on the
Tier-A fixture would not establish generalization.

## Decision

Use a standalone deterministic Python replay over the committed individual
traces. Validate every input hash, fixture boundary, record ordinal, key range,
payload size, schema identity, artifact identity, and M5.1 key adapter before
any simulation starts. The primary comparison is a cold cache reset for every
fixture. Persistent-cache results are reported separately in manifest order
and reverse order; they are directional and do not merge fixture boundaries.

The modeled expert-cache budgets are exact binary 1, 2, 4, 6, 8, 12, 16, 24,
32, and 48 GiB. The budget is the same payload-byte budget accepted by the
runtime `ExpertCache`; dense residency is not subtracted. The 64-byte metadata
and 4,096-byte alignment values remain recorded accounting context, but are
not silently charged against payload capacity.

The policy set is:

- strict global LRU, with deterministic oldest-access then key tie-breaking;
- architecture-only layer LRU with equal non-borrowable per-layer quotas;
- corpus-calibrated layer LRU with proportional non-borrowable quotas, marked
  diagnostic/offline-calibrated;
- observed-frequency LFU with per-cache-lifecycle counters and recency/key
  tie-breaking;
- 80/20 segmented LRU with probationary admission and protected promotion;
- offline Belady, retained only as a theoretical upper bound.

All policies support variable payload sizes, exact byte admission, and
deterministic eviction. Every request is counted regardless of hit/miss.

## Resulting policy decision

Retain strict global LRU as the single policy for the next runtime experiment.
LFU has an isolated gain at 6 GiB but is below global LRU at 8 GiB and only
marginally ahead at 12/16 GiB. Layer-aware policies are not consistently
better, and segmented LRU adds complexity without a stable gain. The next
runtime validation matrix is 8 versus 16 GiB global LRU; it is not run in
M5.2-02.

Classify 8 GiB as `useful_for_selected_workloads`, not a universal balanced
preset. In the cold corpus it provides 15.5660% macro and 24.3750% micro byte
hit rate, while two fixtures have zero hits and the worst case is zero. The
mode is useful for the Tier-A control, code, longer context, and longer decode
workloads represented here. 16 GiB is a modeled higher-memory candidate, not
production-ready.

## Consequences

The result separates macro fixture averages from micro request-weighted
results, preserving visibility into short-workload and long-workload variance.
It also records first-hit and 10/25/40/50/75/90% threshold budgets, per-layer
counts, reuse distances, hot-key concentration, prefill/decode scope, and
persistent cross-session hits.

The simulation does not claim token throughput, measured latency, physical
disk I/O, or linear speedup. Belady uses future knowledge and cannot be a
production candidate. Calibrated layer quotas use corpus statistics and are
diagnostic only. The corpus is intentionally small and does not replace later
runtime validation or production traffic coverage.
