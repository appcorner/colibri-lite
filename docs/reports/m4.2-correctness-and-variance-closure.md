# M4.2 Correctness and Numerical-Variance Closure

## Verdict

M4.2 is **completed with documented numerical variance**.

The pinned Qwen3-30B-A3B Rust F32 path is correctness-valid for the frozen
fixture. It converts and loads the intended weights, matches the ordered
internal arithmetic contract, stays within every checkpoint-specific
Transformers-F32 budget, preserves deterministic routing, validates selected
expert intermediates, generates `[1096, 374]`, and reuses a fixed-capacity KV
cache without growth or overwrite.

No unresolved implementation defect remains in the validated path. The current
runtime is not performance-ready: it is scalar, rereads 73 GB of logical
artifact data for the six-position fixture, and has a one-expert cache with no
hits. Those limitations do not invalidate the correctness result.

`ad44e777bcd18fa416d9da3bd8f70d33ebb85d39`. The canonical root manifest is
`f133d733612840ad691d637732d4ef2de1e0242c4bb1d92521b49dfcfb1b8cd2`.

## RMSNorm Reduction Order

Ordered Python F32 and Rust match bit-for-bit for every scalar intermediate and
all 8,192 isolated weighted outputs. Both use divisor 2048, epsilon before the
square root, and `(x * reciprocal_rms) * weight`.

PyTorch F32 first differs at Layer 0 token 0's sum-of-squares reduction. With
identical inputs, its maximum weighted-output difference from ordered/Rust is
`3.5762786865234375e-7`, 5 ULP. The provisional cross-runtime guards remain 8
ULP and `5e-7`, and both must pass. They are not universal runtime tolerances.

The original propagated Layer-0 post-norm maximum was `3.9935112e-6` because
the RMSNorm received a residual already differing by `1.2516975e-6`. Later
post-attention norm errors reached `2.4414e-4` at Layer 46 while remaining
inside layer-specific propagated budgets. Final model RMSNorm errors for steps
3, 4, and 5 were `4.0054e-5`, `2.2888e-5`, and `5.7220e-5`.

These propagated values combine incoming error and local reduction variance;
they are not evidence of a Rust RMSNorm defect. Rust RMSNorm arithmetic and the
frozen exact internal contract remain unchanged.

## Router Semantics and Oracle Roles

PyTorch does not guarantee stable `torch.topk` indices for ties. Rust's frozen
deterministic policy is higher score first, then lower expert ID for equal
scores. This refines runtime determinism without changing router probabilities
or model semantics.

For each token, exact upstream IDs are assertable only when the positive kth-
selected versus highest-unselected margin exceeds the documented bound and at
least twice the measured router-logit error. The classifications are:

- `exact_match_safe`: IDs match and the boundary is safely separated;
- `numerically_ambiguous`: the boundary cannot support an exact assertion;
- `true_mismatch`: IDs differ despite a safe boundary; validation stops.

Transformers F32 using the same BF16-derived weights is the primary numerical
oracle for Rust F32. Transformers BF16 is a separate model-behavior reference.
BF16 differences are not Rust failures unless the separately defined semantic
comparison supports that conclusion.

BF16, F32, and Rust router IDs matched at Layers 0 and 1. All Rust/F32 router
IDs matched through Layer 47. Layer-47 margins ranged from
`0.04711652` to `0.23168182`, versus required margins from `0.00001812` to
`0.00006008`; all were `exact_match_safe`. BF16 propagation began selecting a
different path from F32 at Layer 2 and its Layer-24/47 router classifications
were ambiguous. Even so, BF16 and F32 produced the same two generated token IDs.

## Reference Occurrence Batching

The initial M4.2-03 exporter recomputed a single expert occurrence in isolation.
That changed the matrix operation's batch shape and produced a Layer-0
aggregate discrepancy. Separate versus concatenated gate/up calls were
bit-identical for the same input, ruling out gate/up ordering.

Restoring genuine Transformers occurrence batching for every selected expert
reproduced the frozen Layer-0, Layer-1, and Layer-24 aggregates exactly. On the
Rust side, every test-only traced down projection was bit-identical to the
unchanged normal `expert_mlp` output. The issue is therefore a reference/exporter
problem, not a runtime defect or compensating numerical error.

## Propagated Drift Through Layer 47

The Layer-0 final block error was `1.9073486328125e-6`. Early expert propagation
increased block error to `3.5095e-4` at Layer 1, `7.9346e-4` at Layer 2, and
`2.3193359375e-3` at Layer 3. Layer 3 had the largest local increase,
`1.52587890625e-3`, and the largest absolute block drift.

Layers 4 through 45 formed a stable plateau at `2.3193359375e-3`; there was no
late positive jump. Layer 46's `1.3427734375e-3` MoE difference partially
canceled the incoming maximum and reduced block drift to `9.765625e-4`.
Layer-47 attention reduced residual drift further to `9.1552734375e-4` before
post-attention RMSNorm.

The pattern is bounded early stepwise growth, a long stable plateau, and a late
decrease. It is neither monotonic growth nor an unexplained anomaly, and it did
not change F32 router semantics. Selected Layer-47 generation checkpoints then
observed expert error `1.2255e-4`, combined-MoE error `8.7738e-5`, and final
block error `1.6785e-4` at prompt step 3.

All-logit F32-versus-Rust maxima were `3.3379e-5`, `3.8147e-5`, and
`4.4823e-5` for selection steps 3 through 5. Generated-step top-1 margins were
`0.4280987` and `6.5719471`, versus required margins `6.6757e-5` and
`7.6294e-5`. Generated IDs were exactly `[1096, 374]`, all six F32 top-20 ranks
matched, and NaN/Inf counts were zero. The drift was bounded and
non-semantics-changing for this fixture.

## KV Cache Correctness

The regression uses 48 layers, fixed capacity/final length 6, and per-layer key
and value shape `[6, 4, 128]`. Its 96 fixed payload allocations total
1,179,648 bytes. Positions advance exactly from 0 through 5. Allocations do not
grow, and every prior K/V prefix remains byte-identical after each append.

Transformers F32 incremental versus full recomputation maximum all-logit
differences were `5.7220e-6`, `5.7220e-6`, and `6.1989e-6` at lengths 4, 5, and
6. These remain diagnostic observations, not a newly promoted blanket
tolerance. Incremental, recomputed, and Rust cached argmax results agree.

Any future cache change must rerun fixed-allocation, prefix-overwrite,
incremental/recompute, router, intermediate, generation, bounded-memory, and
counter-reconciliation regressions.

## Remaining Correctness Risks

- Full-model generation coverage is one compact prompt and two generated
  tokens; long-context and broad-prompt behavior remain unvalidated.
- Intermediate expert evidence is representative rather than exhaustive.
- The 8-ULP/`5e-7` PyTorch RMSNorm guard has measured platform headroom but is
  still provisional outside the pinned environment.
- Quantization, SIMD, mmap, prefetch, alternative accumulation, and parallel
  expert execution have no correctness evidence yet.
- BF16 propagation is different model behavior and must not be mistaken for an
  F32 runtime guarantee.

There is no known unresolved correctness failure in the validated scalar Rust
F32 fixture.
