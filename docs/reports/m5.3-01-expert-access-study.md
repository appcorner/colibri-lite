# M5.3-01 Expert access study

Status: complete for review

This task measured the existing expert-loading path and simulated read-range
grouping. It did not change `ExpertStore`, strict global LRU, cache budgets,
artifact layout, numerical kernels, or the production reader implementation.
No production mmap path was implemented.

## Current load path

On an expert miss, the current path is:

```text
ExpertStore name lookup
  -> strict global-LRU miss/admission
  -> ArtifactReader manifest lookup and path canonicalization
  -> File::open
  -> metadata
  -> seek to payload offset
  -> fresh Vec<u8> allocation
  -> read_exact
  -> SHA-256 verification
  -> Vec<u8> returned and converted to Arc<[u8]>
  -> F32 gate/up/down decode allocations
  -> expert MLP computation
```

The default reader still opens and closes the file for every tensor read. The
M5.3 feature `m5-3-instrumentation` adds runtime-only counters and timing
estimates without changing the default build. The read counter is at the Rust
`Read::read` boundary; it is not an OS kernel-syscall counter. Timing fields
are operational evidence and are not included in deterministic trace content.

`ExpertPathMetrics` separates the successful store request wall time into an
instrumented expert-load interval and the remaining cache lookup/admission
interval. The latter includes cache bookkeeping, eviction, and lease/Arc work;
it is not presented as a pure hash-table timing.

## Artifact layout

The expert artifact has one shard per selected decoder layer, 48 shards total,
and 128 experts per shard. Every expert payload is one contiguous
18,874,368-byte range. The three projections are each 6,291,456 bytes and are
laid out without gaps:

```text
gate [0, 6,291,456)
up   [6,291,456, 12,582,912)
down [12,582,912, 18,874,368)
```

Expert payloads are ordered by expert ID within a shard. The payload stride is
18,874,368 bytes, the shard size is exactly 128 strides, and all payload starts
are 4096-byte aligned. Adjacent expert IDs in one layer therefore form
exactly adjacent ranges. The selected expert IDs in the traces are usually
not adjacent: the same-layer adjacent-gap median is 169,869,312 to
188,743,680 bytes in the selected workloads, depending on workload.

## Range-grouping simulation

The deterministic tool `scripts/analyze_m5_3_expert_ranges.py` replayed the
authoritative miss sequence under a one-expert cache, 8 GiB LRU, and 16 GiB
LRU. It evaluated no coalescing, exact adjacency, 4 KiB, 64 KiB, 1 MiB, and
one-layer-batch grouping. Useful bytes are the requested miss payload bytes;
over-read is the range-group byte count minus useful bytes.

| Fixture | Requests | Unique keys | 8 GiB misses | 16 GiB misses | Exact-adjacent reads / no-coalesce reads | 1 MiB reads | Layer-batch reads / amplification |
|---|---:|---:|---:|---:|---:|---:|---:|
| Tier-A control | 2,304 | 1,332 | 1,585 | 1,373 | 1,528 / 1,585 | 1,528 | 288 / 14.94x |
| Short Thai | 384 | 384 | 384 | 384 | 359 / 384 | 359 | 48 / 12.71x |
| Code | 1,536 | 1,099 | 1,208 | 1,101 | 1,150 / 1,208 | 1,150 | 192 / 14.20x |
| Long context | 3,456 | 1,665 | 2,353 | 1,898 | 2,248 / 2,353 | 2,248 | 430 / 15.53x |
| Long decode | 1,920 | 1,224 | 1,336 | 1,238 | 1,280 / 1,336 | 1,280 | 239 / 14.96x |

The exact-adjacency operation reduction was 3.5–6.5% in this matrix and had
zero byte amplification. Thresholds through 1 MiB produced the same grouping
as exact adjacency because the non-adjacent selected ranges have much larger
gaps. One-layer reads reduce operation count substantially but read roughly
12.3–15.8 times the useful payload, so they are not a reasonable default
coalescing strategy.

This is a grouping model, not a latency model. It does not claim that fewer
read calls translate linearly to faster execution.

## Instrumented storage measurements

The isolated release benchmark used the canonical layer-47 artifact plan and
SHA-256-checked every payload. It measured the current reader for repeated,
deterministically randomized, and 64-request authoritative miss subsets. The
filesystem and OS page-cache state was uncontrolled; these are not cold-cache
results.

| Current-reader case | Requests | Wall time | Opens | Seeks | Read calls | Allocated/copied bytes | Hash time |
|---|---:|---:|---:|---:|---:|---:|---:|
| Same expert repeated | 3 | 0.290 s | 3 | 3 | 3 | 56.6 MiB | 0.262 s |
| Deterministic randomized experts | 32 | 3.162 s | 32 | 32 | 32 | 604.0 MiB | 2.836 s |
| Authoritative layer-47 miss subset | 64 | 6.410 s | 64 | 64 | 64 | 1.125 GiB | 5.748 s |

For the 64-request current-reader case, the measured read interval was
0.543 s, open time 3.25 ms, metadata 0.534 ms, and seek time 0.067 ms. The
reader performed 64 fresh allocations and 64 payload copies. The current
reader's measured SHA-256 interval was 5.748 s, approximately 90% of the
observed wall time in this run. This is the cost of the required integrity
check in the measured path, not a claim about physical storage latency.

An isolated persistent-handle loop over the same ranges took 6.367 s with a
fresh buffer per range and 5.996 s with one reusable buffer. The one-run
difference is directional, not a benchmark claim. It indicates that buffer
reuse is a more measurable low-risk signal than persistent open/close on this
machine. The current path's open and seek timings are small relative to hash
and payload processing.

The ExpertStore instrumentation over the same 64 miss requests measured:

* cache requests/hits/misses/loads/evictions: 64 / 0 / 64 / 64 / 63;
* expert bytes read: 1,207,959,552;
* total store path: 7.131 s;
* instrumented expert-load interval: 6.511 s;
* remaining cache lookup/admission/lease interval: 0.620 s;
* reader file opens, seeks, read calls, and allocations: 64 each.

The benchmark verifies payload hashes in both the current reader and isolated
loops. It does not measure matrix computation, full-model decode construction,
or physical disk I/O. Those categories remain unknown in this study rather
than being attributed to the file read.

## Mmap feasibility

Windows x64 provides native file-mapping primitives through
`CreateFileMappingW` and `MapViewOfFile`. A full-shard mapping would expose
approximately 115.96 GiB of virtual address space across 48 files; that is not
the same as resident or committed memory and would not fit within an expert
payload budget as resident data. Subrange mappings require Windows allocation
granularity handling, mapping/file-handle lifetime management, and careful
unmapping on all error paths.

The current `Vec<u8>`/`Arc<[u8]>` lease contract also means a mapped slice would
need a new ownership contract before it could safely outlive a mapping handle.
Mapping could reduce opens, seeks, and one payload copy, but the current
streaming path subsequently decodes the three projections into F32 vectors.
The measured path is dominated by payload verification and the range workload
does not show broad near-contiguous selected ranges. No evidence here justifies
introducing an unsafe Windows mapping boundary or a mapping dependency.

Mmap remains feasible as a separately reviewed experiment, but it is not
selected for the next prototype.

## Decisions

1. File reopening is present and measurable in counters, but not material in
   this controlled run relative to hash and payload processing.
2. Read calls are one per current miss. Exact-adjacent grouping can remove a
   small number of operations, while bounded gaps through 1 MiB add no further
   benefit in the selected matrix.
3. Allocation/copy is a plausible low-risk refinement; the isolated reusable
   buffer run was faster than its fresh-buffer counterpart, but requires a
   repeated controlled benchmark before a performance claim.
4. Broad layer-batch coalescing is rejected for now because of severe byte
   amplification.
5. Mmap is not selected because its ownership/unsafe complexity is not
   supported by measured benefit in this task.
6. The larger expert cache reduces miss frequency, so storage refinement is
   most relevant to the remaining miss-heavy Thai and lower-cacheability cases.
7. Scalar matrix compute versus storage remains unclassified because this task
   deliberately measured the storage path in isolation. It is not valid to
   call scalar compute dominant from these measurements.

The selected next storage-access prototype is **reusable aligned read
buffers**. It is the smallest reversible change that targets an observed
allocation/copy cost, preserves the current artifact and reader contract for
an isolated comparison, and supplies diagnostic value before considering
persistent handles or mmap. It is not implemented in M5.3-01.
