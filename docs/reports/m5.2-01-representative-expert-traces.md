## Correctness and repeatability

All eight accepted traces were captured twice. Every repeat has identical
request count, ordered layer/expert sequence, generated IDs, non-timing
counters, canonical bytes, and SHA-256. The frozen Tier-A control is the
committed M5.1-00 trace and its two repeat entries are the same canonical
hash.

Per-fixture checks passed for finite output state, deterministic generated IDs,
valid layer/expert ranges, contiguous global ordinals, exact payload sizes,
KV-cache capacity/final sequence length, cache accounting, and zero
oversized-entry or blocked-eviction events. The five reused Tier-B fixtures
retain their committed M4.3 F32 evidence references. The longer-context case
has Rust deterministic regression evidence; the longer-decode case is
trace-only and makes no Transformers-equivalence claim.

The existing simulator record adapter accepts every new v2 record through its
`key_of` contract. The M5.1 cache simulator itself was not run on this corpus,
as required.

## Comparison with frozen Tier-A

The short Tier-B language, repeated, and formatting cases are much shorter
than Tier-A and have mostly unique requests: unique-key ratios range from
`0.947917` to `1.0`, repeated-key percentages from 0% to 5.2083%, and no
decode reuse. The code case is also shorter but has more reuse than those
cases (`28.4505%`). The longer-context case is the most reuse-heavy workload
(`51.8229%`) and has a longer maximum reuse distance (3,075 versus 1,924).
The longer-decode case contains 1,152 decode occurrences and 259
cross-phase reuses, while its repeat rate is below Tier-A (`36.25%` versus
`42.1875%`).

Across the corpus, the original Tier-A trace is therefore neither a universal
upper nor lower bound: short natural-language and formatting traces are more
optimistic about reuse distance/cacheability, while the longer context is
more reuse-heavy and has a longer tail. Descriptive classification of the
existing 8 GiB recommendation: **inconclusive**. No new cache policy or RAM
preset is recommended by this task; M5.2-02 is required for that decision.

## M5.1-03 counter discrepancy follow-up

The machine-readable diagnosis is
`models/qwen3-30b-a3b/m5.2-01-m5.1-03-counter-discrepancy-v1.json`.

| Budget semantics | Hits / loads / evictions |
|---|---:|
| M5.1-03 exact nominal 8 GiB runtime | 719 / 1,585 / 1,130 |
| M5.1-02 usable 8 GiB replay budget `8,212,550,504` | 719 / 1,585 / 1,150 |
| M5.1-03 exact nominal 16 GiB runtime | 931 / 1,373 / 463 |
| M5.1-02 usable 16 GiB replay budget `16,802,485,096` | 929 / 1,375 / 485 |

The cause is confirmed as payload-budget accounting. M5.1-02 charged modeled
metadata, alignment, and safety overhead before assigning payload bytes;
M5.1-03 used the exact nominal payload budgets. The ordered request count,
strict global-LRU semantics, sequential lease boundary, cache-update behavior,
and instrumentation boundary agree. Exact nominal-budget replay matches the
full-model counters, so no runtime behavior defect or instrumentation fix was
needed.
