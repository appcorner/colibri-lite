# M5.3-04 isolated read-only mmap expert-access prototype

Status: complete for review; experimental reader rejected for runtime adoption.

## Scope and design

This task evaluated an opt-in read-only mmap path for expert shards only. Dense
weights, tokenizer files, manifests, KV state, cache policy, artifact layout,
and numerical kernels were unchanged. The reference seek/read reader remained
the default; the reusable staging reader was benchmark context only.

The canonical artifact was validated at
`D:\models\colibri-lite\qwen3-30b-a3b\artifact-v1`:

* root SHA-256: `f133d733612840ad691d637732d4ef2de1e0242c4bb1d92521b49dfcfb1b8cd2`;
* 48 expert shards, each 2,415,919,104 bytes;
* packed expert payload: 18,874,368 bytes;
* model/revision: Qwen/Qwen3-30B-A3B / `ad44e777bcd18fa416d9da3bd8f70d33ebb85d39`.

The isolated `clr-mmap` crate uses `memmap2 = 0.9.11` (MIT OR Apache-2.0,
Rust 1.65, Windows-compatible). It lazily maps each complete expert shard
read-only and retains its file handle and mapping until `ArtifactReader` drop.
The exact mapped range is SHA-256 checked and copied into owned expert bytes
before cache admission. Therefore this prototype is not zero-copy expert
construction. Mappings are bounded by reader lifetime, but the initial
strategy has no mapping eviction.

The only unsafe operation is the minimal `MmapOptions::map` call in
`clr-mmap`; its invariants are documented: read-only file, offset zero, exact
metadata length, joint file/map ownership, shared slices only, and no mutable
API. The production workspace remains `unsafe_code = "forbid"`.

## Correctness and lifecycle gates

The mmap adapter and storage tests covered exact ranges, final-byte and
unaligned ranges, empty/invalid/truncated/missing files, two-shard reuse,
repeated reads, reader drop followed by Windows file deletion, and exact
reference byte/hash equivalence. `cargo test -p clr-mmap` passed 2 tests;
`cargo test -p clr-storage --features m5-3-mmap` passed 24 tests.

The full runtime matrix preserved exact traces, generated IDs, router/expert
order, cache counters, logical expert bytes, KV invariants, finite outputs,
and bounded payload residency. All 16 simulation comparisons were exact; all
oversized-entry and blocked-eviction counters were zero.

## Microbenchmark

The benchmark used identical hash-checked ranges, two repetitions, and
uncontrolled filesystem cache state. It covered repeated same-expert,
randomized, authoritative 64-miss, same-shard clustered, and cross-shard
access. For the authoritative 64-miss subset:

| Path | Mean time | Opens | Read calls | Allocation behavior |
|---|---:|---:|---:|---|
| reference | 5.827 s | 64 | 64 | 64 payload allocations |
| reusable staging | 5.792 s | 64 | 64 | 1 staging allocation |
| mmap | 6.053 s | 1 mapping | 0 | 64 owned payload copies |

Mmap first-touch was about 89--91 ms for one shard and 171--177 ms for the
two-shard isolated case. It reduced syscall/open accounting, not logical
payload copy or SHA-256 verification. Benchmark SHA-256:
`87bfdbdd44975096e20a7d59c3fc6b584e820aed4370d88d62d1ac335eb2b1cb`.

Mmap was slower in all 8 paired comparisons. Median paired change was
`+5.92%`. It did not change expert loads, hits, evictions, logical expert
bytes, generated IDs, or request order. Results JSON SHA-256:
`05a5aff20b5ce7698825ff1cb50bddc7394d02d54a7673e89382a9a31547af64`.

Each full-shard mapping run reported 108 GiB mapped virtual bytes for the
selected workloads. This is not resident RAM. Mmap peak working set was
29.46--39.00 GiB; reference peak working set was 8.72/17.31 GiB at 8/16 GiB
cache modes. Peak sampled private bytes for mmap were approximately 8.96 GiB
at 8 GiB and 17.57 GiB at 16 GiB. Page-fault counters and physical disk I/O
were not controlled or claimed.

## Decision

Classification: `insufficient_runtime_value`.

Mmap is technically correct and diagnostically useful, but not an accepted
storage primitive: median full-runtime change was a 5.92% regression, every
paired comparison regressed, and mapped virtual/working-set behavior is more
complex. The reference reader remains default; mmap is not promoted or
exposed as an optional production mode.

The selected next task is **stop the current storage-access optimization path
due insufficient runtime value**. Do not start another optimization in this
task. Any later storage proposal needs a new measured design and review.

## Reproduction and limitations

The deterministic capture tool is `scripts/capture_m5_3_04_mmap.py`; the
storage benchmark is `m5_3_expert_access_bench` with reusable and mmap
features. The full-runtime command was:

```powershell
cargo test -p clr-qwen3-moe --release --features full-model-validation,m5-3-mmap,m5-3-compute-profiling --no-run
python scripts/capture_m5_3_04_mmap.py --test-executable target\release\deps\clr_qwen3_moe-db3bd54b8b0b108b.exe --run-root .m5-3-04-runs --run-id matrix-fda29de
```

The committed M5.2 fixture/corpus definitions supplied exact token IDs and
generation settings. This task did not measure cold filesystem state,
controlled page faults, physical disk I/O, or a production bounded mapping
cache, and it makes no throughput claim.
