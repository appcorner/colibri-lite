# M5.3-03 Compute profiling

## Status and decision

M5.3-03 is complete for review. The feature-gated profiler preserved the
reference reader, strict global LRU, numerical execution, request order, and
bounded payload residency. Eight detailed full-model profiles and three Tier-A
profiling-mode runs passed the deterministic non-timing checks. The measured
path is storage-bound, with scalar expert computation and the LM head as
secondary costs. No compute kernel was changed.

The next selected prototype is an **isolated read-only mmap expert-access
prototype**. It is a storage-path diagnostic only and is not implemented by
this task. It must remain outside the default ExpertStore path until the
Windows mapping, lifetime, working-set, and byte-equivalence gates are
reviewed.

## Profiling design

The profiler is enabled with the
`m5-3-compute-profiling` feature, which includes the existing storage
instrumentation but leaves the reference allocated reader as the active mode.
It uses safe Rust and a thread-local collector. `Instant` provides monotonic
timing. A scope stack records inclusive and exclusive duration, call count,
minimum, maximum, and total nanoseconds. Events are serialized in deterministic
`(phase, layer, operation)` order; timing fields are evidence metadata and are
not part of the non-timing correctness fingerprint.

The schema is
`colibri-qwen3-moe-m5.3-03-compute-profile-v1`. Profiled phases are
`prefill`, `decode_1`, `decode_2`, and subsequent decode phases. Instrumented
scopes cover embedding, decoder layers, norms, Q/K/V/O projections, attention,
router/top-k, cache lookup and expert load, expert payload decoding, expert
gate/up/activation and down projection, residual/KV operations, final norm,
LM head, and token selection. Linear operations also record dimensions,
estimated FLOPs, input/weight/output bytes, and the scalar implementation.

The profile is hierarchical. Parent scopes are not added to child totals when
calculating an exclusive operation ranking. The model-total event is compared
with the runtime timing boundary; the ratio was 0.99988–1.00043 for the
detailed rows. Small residuals are measurement-boundary and uninstrumented
runtime work, not attributed to storage or compute without evidence.

The host and build context are the M4 baseline: Windows 11 Enterprise,
Intel Core i7-1165G7, 8 logical processors, approximately 51.25 GB installed
RAM, Rust 1.96.1, `x86_64-pc-windows-msvc`, release build, F32 compute and KV
cache, and eight configured runtime threads. Filesystem cache state was
uncontrolled. No external sampling profiler was required or used.

### Top operations

Across all detailed rows, the stable exclusive-time ranking was:

1. `model.total` as the accounting boundary;
2. `experts.total`;
3. `cache.lookup_and_expert_load`;
4. `expert.mlp.total`;
5. `expert.payload_decode`;
6. `lm_head`;
7. `expert.gate_up_activation`;
8. decoder-layer and attention scopes.

The ranking is intentionally interpreted with hierarchy in mind. The
`experts.total` and cache path include the load-side work; the separately
reported MLP scope does not include that load time. `expert.payload_decode`
was approximately 4.2–5.1% of total. LM head was approximately 2.8–3.8%,
attention totals approximately 2.0–2.8%, and expert MLP approximately
4.1–5.5%. The `expert.gate_up_activation` operation includes the activation
inside the existing gate/up path; the evidence does not claim a separate
gate-only timing.

### Prefill versus decode

Long context is prefill-dominant: decoder layers account for approximately
298.1 s at 8 GiB and 256.9 s at 16 GiB, with LM head around 10.4 s. Long
decode has approximately 86–89 s of prefill and 75–86 s of decode; its three
decode layer totals were about 27–29 s, 25–29 s, and 20–25 s depending on
budget. Tier-A has approximately 149–153 s of prefill decoder work and
24.8–29.9 s per decode step. The measured storage path remains the dominant
component in both phases for these workloads, although the long-context
profile makes the cache benefit most visible.

The LM head is a major named operation relative to attention, but not the
dominant whole-runtime bottleneck in this matrix.

## Storage, allocation, and unclassified work

The reference reader storage evidence confirms one allocation, open, seek, and
read per expert miss in the profiled rows. For example, Tier-A 8 GiB had 1,585
loads, 1,585 file opens, 1,585 seeks, 1,585 read calls, and 29,915,873,280
payload bytes allocated/read. The measured storage reader split was roughly:

* payload read: 14.6–30.4 s;
* payload SHA-256 verification: 87.3–187.9 s;
* cache lookup: 0.23–2.68 s;
* expert-load path outside those subscopes: the remainder of the path.

This instrumentation does not claim that the hash interval is physical disk
I/O; it measures the current logical reader path and verification work. It
also does not isolate the general-purpose allocator or every tensor copy as a
separate profiler scope. The storage metrics do record the reference reader's
allocated bytes and copied bytes; these counters are retained in the
machine-readable evidence. Allocation/copy therefore remains an observed
component, but its percentage is **unknown rather than inferred** from total
runtime. No timing is assigned to dense-weight loading versus dense compute by
this profile because the existing dense reader does not expose a sufficiently
separate boundary. The residual between model total and the explicitly
instrumented scopes is reported by the profile ratio and is not silently
classified.

## Bottleneck classification and candidate review

The runtime is classified as:

* **storage-bound**: the cache lookup/expert-load path is 71.6–76.4% of the
  measured model profile, and payload verification alone is larger than expert
  MLP compute;
* **mixed scalar-compute secondary**: expert MLP is 4.1–5.5%, with Q/O
  projections, LM head, attention, and other scalar operations forming the
  remainder;
* **not established as allocation/copy-bound**: allocation/copy counters exist,
  but no isolated timing fraction was measured;
* **not LM-head-bound or attention-bound** for this workload matrix.

Candidate assessment:

| Candidate | Affected evidence | Complexity / risk | Decision |
| --- | --- | --- | --- |
| Threaded expert matmul | roughly 4–5.5% expert MLP share | high scheduling and reproducibility surface | not selected now |
| SIMD expert matmul | same limited share | numerical-order, portability, and unsafe-kernel risk | not selected now |
| Optimized LM head | roughly 2.8–3.8% | large vocabulary and numerical-order risk | not selected now |
| Fused expert MLP | gate/up/activation/down subset | fusion/layout and numerical-order risk | not selected now |
| Resident dense weights | dense path not isolated here | changes memory dimension and scope | not selected now |
| Additional storage work | 71.6–76.4% load path | mmap lifetime/page-fault/working-set risk | selected as isolated mmap study |

The larger cache reduces miss frequency and storage time, but it does not
change the conclusion that storage-path work has the largest measured affected
share. The reusable staging-buffer result remains diagnostic only; it is not
promoted because its full-runtime matrix regressed in 9 of 12 paired
comparisons.

## Selected prototype and performance gates

The selected next task is an isolated, read-only mmap expert-access prototype.
It must not replace `ExpertStore` in this task's follow-up until reviewed. The
prototype should compare the exact same expert ranges against the reference
reader and measure read, verification, copy, mapping, virtual bytes, resident
working set, and committed/private memory separately. It must remain
feature-gated and opt-in.

Minimum gates for retaining the prototype for further review are:

1. exact payload bytes, tensor shapes, sampled/full payload hashes, generated
   IDs, router decisions, request order, cache counters, and KV behavior;
2. no change to global LRU, cache budgets, lease ownership, artifact layout,
   or arithmetic order;
3. documented Windows file and mapping-handle lifetime, replacement/deletion,
   alignment, unmapping, and error behavior;
4. separate reporting of virtual mapped bytes, resident working set,
   committed/private bytes, logical reads, and physical-I/O unknowns;
5. no unbounded mapping or buffer growth and no unsafe boundary unless a safe
   implementation is impossible and a separate review is completed;
6. a material reduction in the measured reader-path bottleneck without an
   end-to-end regression outside an explicitly documented noise range before
   any default-reader discussion.

No token/s projection or physical-disk saving is implied by this selection.
