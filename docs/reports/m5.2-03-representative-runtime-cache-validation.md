# M5.2-03 Representative Full-Runtime Cache Validation

## Result

The exact binary 8 GiB and 16 GiB strict global-LRU budgets were validated on
six representative Qwen3-30B-A3B workloads. All 18 requested runs completed:
Tier-A, long-context, and long-decode were repeated twice at each budget; the
other three fixtures were run once at each budget.

The runtime counters matched the committed M5.2-02 simulation exactly for
every fixture and budget. Generated IDs, router/request order, KV-cache
invariants, bounded payload residency, and finite-output checks passed. The
cache remains an `accepted_with_workload_limitations` memory primitive:
additional residency is valuable for repeated long-context and control
workloads, but it has no cache benefit for the selected one-token Thai and
special-token workloads.

The M5.2-02 classification is validated with a narrower scope:

- 8 GiB is `useful_for_selected_workloads` and requires a workload-aware
  recommendation; it is not a universal balanced preset.
- 16 GiB is `useful_for_cacheable_workloads` and also requires workload-aware
  selection.
- The selected runtime policy remains strict global LRU.

No throughput or physical-I/O claim is made. Filesystem cache state was
uncontrolled, and timing is directional evidence only.

## Simulation versus runtime

All 18 records carry `simulation_comparison.classification: "exact"` in the
machine-readable result. The compared fields were requests, hits, misses,
loads, evictions, expert bytes loaded, expert bytes avoided, and peak resident
payload bytes. There were no explained-accounting or unexplained mismatches in
this exact-budget replay.

This also resolves the M5.1-03 follow-up. The earlier M5.1-02 replay charged
modeled cache overhead before assigning payload capacity: 8 GiB had a usable
payload budget of 8,212,550,504 bytes and 16 GiB had 16,802,485,096 bytes.
The full runtime and the M5.2-02 expectations use the nominal payload budgets
without that modeled metadata charge. That accounts for the earlier 8 GiB
`1150` versus `1130` eviction difference and the earlier 16 GiB
`929/1375/485` versus `931/1373/463` hit/load/eviction difference. The
authoritative request boundary and lease lifetime were not defective; no
runtime cache semantic change was required.

## Preset and policy decisions

The following are documentation-only modeled modes, not production presets:

| Mode | Expert budget | Policy | Modeled full-corpus macro / micro | Intended use |
|---|---:|---|---:|---|
| Ultra-low-memory streaming | 1 GiB | global LRU | 0% / 0% | bounded-memory execution where rereads are accepted |
| Low-memory cache | 4 GiB | global LRU | 0% / 0% | transitional low-memory mode; not useful for this corpus |
| Workload-selective balanced | 8 GiB | global LRU | 15.5660% / 24.3750% | control, code, long-context, and long-decode workloads |
| Performance-oriented | 16 GiB | global LRU | 19.8707% / 31.9444% | cacheable long-context/control workloads when RAM permits |

No automatic sizing is implemented. The runtime should expose explicit byte
configuration only. A future workload-aware design may select 8 or 16 GiB from
known workload characteristics, but both choices require further validation.

The single policy decision is: **retain strict global LRU**. M5.2-02 showed
architecture-only layer LRU below global LRU at both 8 and 16 GiB; calibrated
layer LRU is diagnostic and not generalizable. LFU and segmented LRU were near
global LRU but changed ranking across budgets, while their calibration/admission
complexity and lack of runtime evidence do not justify a production policy
change. Belady remains an offline upper bound only.

## Performance-value classification and next task

The cache primitive is `accepted_with_workload_limitations`. It is correct and
bounded, and it materially avoids expert logical reads for repeated workloads.
The value is not universal: two selected workloads had zero hits at both
budgets, and process memory approximately doubled when the cache filled from
8 to 16 GiB.

The selected next task is **mmap/coalesced expert access study**. It is selected because
the 16 GiB runs still loaded 20.8–35.8 GiB of expert payload for the cacheable
workloads and expert logical reads remain the largest measured variable in the
matrix. This report does not start that study, change access semantics, or
make a physical-storage claim.
