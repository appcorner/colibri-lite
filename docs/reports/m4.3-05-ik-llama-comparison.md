# M4.3-05 ik_llama Comparison

## Result

This task completed as an external benchmark and gap analysis. The pinned
`ik_llama.cpp` CPU build loaded the exact Qwen3-30B-A3B architecture in Q4_K_M
format. It is a directional reference, not a quality-equivalent replacement
for the frozen colibri F32 baseline.

## Environment and comparability

Repository URL, commit, build switches, compiler, CPU features, model revision,
file hash, and runtime settings are recorded in
`models/qwen3-30b-a3b/m4.3-05-ik-llama-environment-v1.json`. The Q4 file is
18,556,685,824 bytes (4.861 BPW); the canonical colibri artifact is
122,147,666,917 component bytes and F32 storage. Both runs used Windows CPU,
8 threads, context 128, greedy sampling, and the exact four-token fixture
`Hello, world!` -> `[9707, 11, 1879, 0]`. The ik_llama build had mmap, fused MoE,
Flash Attention, and OpenMP enabled; colibri used its validated scalar,
on-demand expert path.

## Same-hardware measurements

| Runtime/measurement | Prompt tok/s | Decode tok/s | Startup wall (load + run) | Peak working set | Peak private |
| --- | ---: | ---: | ---: | ---: | ---: |
| colibri M4.2-05 mean of A-C | 0.0147 | 0.0146 | 2.40 s init (separate) | 145-148 MB | 143 MB |
| ik_llama CLI short fresh runs | 2.963-3.291 | 5.276-5.863 | 5.641-6.224 s | 4.36-4.41 GB | 338-339 MB |
| ik_llama llama-bench, 3 reps | 3.968 +/- 0.875 | 2.819 +/- 0.099 | not exposed | not exposed | not exposed |
| ik_llama 32-token decode | 3.202 | 3.511 | 15.767 s | 9.03 GB | 339 MB |

The short CLI output was `This is` for two generated tokens. The long run was
used only to reduce timer noise; its full text is retained in the raw temporary
run, not in the repository. Model-load milliseconds are not exposed by this
CLI build, so startup wall time includes load and inference. `llama-bench`
prompt and generation numbers are separate tests and should not be combined.

Using the M4.2-05 mean as a directional denominator, llama-bench is about 270x
prompt and 192x decode faster. This is not a claim that quantization alone
causes that ratio: the comparison also changes weight residency, kernels,
threading, batching/graph reuse, and MoE scheduling.

## Memory and I/O

ik_llama reports a 17.277 GiB model tensor buffer, 24 MiB F16 KV cache, 0.58
MiB output buffer, and 76.19 MiB compute buffer. Sampled process working set
was 4.36-4.41 GB for short runs and 9.03 GB for the 32-token run. Colibri's
M4.2-05 process peak was 145-148 MB with 127,823,000 modeled explicit bytes.
The two memory numbers describe different policies and are not interchangeable.

Neither runtime exposed trustworthy physical device-read bytes for this run.
Colibri's logical application reads were 73,004,834,816 bytes (2.428603x
amplification); ik_llama's mmap path did not expose logical read counters. The
Q4 file size is therefore not reported as physical I/O.

## Architecture gap analysis

| Factor | Evidence status | Interpretation |
| --- | --- | --- |
| Quantized model size | directly measured | 18.56 GB Q4 file versus 122.15 GB colibri F32 components reduces memory traffic. |
| mmap/page cache | directly measured/configured | ik_llama mmap is enabled; resident pages grow from short to long run. |
| SIMD kernels | supported by instrumentation | AVX/AVX2/FMA/AVX-512 are enabled; colibri remains scalar. |
| Thread parallelism | directly measured | ik_llama and colibri comparison used 8 host threads, but colibri expert path is sequential. |
| Fused MoE/expert scheduling | supported by instrumentation | ik_llama fused MoE and active-expert scheduling are enabled. |
| Graph reuse and Flash Attention | supported by instrumentation | enabled in ik_llama; absent from the scalar colibri path. |
| Prompt batching | inferred/limited | llama-bench uses a batched benchmark harness; colibri M4.2 prefill is sequential. |
| Memory bandwidth | unknown | no bandwidth counter was collected. |
| LM-head implementation | unknown | no isolated attribution was possible. |

## Decisions

1. The hardware is capable of useful Qwen3-30B-A3B CPU decode speed with an
   optimized runtime.
2. The largest actionable colibri gap is measured memory/I/O and scalar expert
   execution, not a correctness defect. The first experiment should measure a
   resident F32 dense/selected-expert or buffered-I/O design, with no semantic
   changes.
3. Q4 is the practical size/performance frontier demonstrated here; it is not
   accepted as a colibri format. The M4.3-04 per-output-channel INT8 candidate
   remains `quality_risk`.
4. Colibri's low-memory differentiation remains valuable: its process working
   set is roughly two orders of magnitude below the optimized Q4 reference,
   at the cost of extreme latency.
5. A useful operating mode needs at least the measured 18.56 GB Q4 model plus
   runtime headroom; colibri's correctness path operates under its bounded
   127.8 MB modeled explicit budget but is not performance-ready.

## Limitations

The filesystem cache was not dropped or verified. Storage-device metadata and
physical I/O were unavailable under the sandbox. CPU utilization, thermal
frequency, and memory-channel data were not instrumented. No model-quality or
perplexity equivalence claim is made. The Q4 artifact and colibri F32 artifact
are not bitwise comparable.
