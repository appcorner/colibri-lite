# M5.3 phase closure and decision review

Status: review closure after M5.3-04. This report consolidates recorded
evidence; it does not authorize a new runtime optimization.

## Status by decision area

| Area | Classification | Evidence-based status |
| --- | --- | --- |
| Research / feasibility | PASS | Qwen3-30B-A3B F32 executes all 48 layers and generates tokens. |
| Full-model correctness | PASS with documented numerical variance | Frozen F32 checkpoints, deterministic IDs/traces, routing, cache, KV, and finite-output checks passed in the recorded full-model evidence. |
| Low-memory feasibility | PASS | Expert streaming and byte-budgeted cache behavior passed; strict global LRU remains selected. |
| M5.1 cache prototype | Accepted with limitations | Exact trace/runtime counters passed, while workload coverage and uncontrolled filesystem timing remain limitations. |
| M5.2 representative validation | Complete for review / accepted with workload limitations | Simulation and 18 full-runtime runs support exact 8/16 GiB strict-global-LRU behavior, not a universal preset. |
| Reusable buffer | Diagnostic / microbenchmark only | Allocation reduction did not establish end-to-end value; it remains non-default. |
| mmap reader | Rejected for production | Correctness gates passed, but every paired runtime comparison regressed and working set increased materially. |
| Production performance | NOT READY | The runtime is correctness-valid, but no storage-access candidate has established production-performance value. |

## Consolidated evidence

- M0--M4 are complete. M4 established the reproducible F32 baseline with
  documented numerical variance, and the full Qwen3-30B-A3B path executes all
  48 layers and generates deterministic tokens.
- M5.1 and M5.2 recorded passing deterministic behavior, expert streaming,
  exact cache-accounting/replay evidence, and bounded byte-budget cache
  behavior. M5.1-03 and M5.2-03 retain their stated workload, sampling, and
  filesystem-cache limitations.
- M5.3-03 profiles the cache lookup/expert-load path at approximately
  71.6--76.4% of model profile time. This is a bottleneck observation, not an
  acceptance of a storage implementation.
- M5.3-02 reusable-buffer evidence has isolated microbenchmark value only; it
  must not become the default reader.
- M5.3-04 mmap evidence passed byte-equivalence, lifecycle, deterministic,
  cache, KV, and configured payload-budget checks. Mmap was slower in all eight
  paired full-runtime comparisons, with median regression approximately +5.92%.
  Its measured peak working set was approximately 29.46--39.00 GiB; mapped
  virtual bytes and working set are documented separately from resident-budget
  semantics.

## Decisions

- Default reader: retain the reference reader.
- Cache policy: retain strict global LRU.
- Reusable buffer: diagnostic/microbenchmark only; not a production default.
- mmap: rejected for runtime adoption; no mapping-cache follow-up is selected.
- Current storage-access optimization path: STOP.
- Production-performance Go/No-Go: **NO-GO / NOT READY**.

## Proposed future decision gates

These are proposed gates for a separately reviewed candidate; they are not
existing acceptance requirements and do not authorize implementation.

1. Frozen F32 correctness evidence must not regress.
2. The configured cache and memory budget must not be exceeded.
3. Benchmarks must be repeatable.
4. Improvement must be measured end-to-end, not only in microbenchmarks.
5. Working-set and memory semantics must remain explicit.
6. Quantized runtime, GPU, server, web UI, and other deferred features must
   not begin under this direction.

## Recommended direction

Do not start another mmap or storage-access optimization immediately. If review
approves further work, consider resident dense weights plus strict global LRU
as a candidate, beginning with simulation and measurement before any runtime
code change. Require repeatable end-to-end improvement while preserving
correctness and memory limits. If no candidate passes these gates, freeze the
project as a research runtime rather than expand scope without evidence.
